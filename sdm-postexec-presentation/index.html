Sequencer-Defined Metering
& the PostExec mechanism

What SDM is, how the 0x7D transaction works, what landed on develop, which hardfork it ships with, how it's tested, and what's next.
Presenter Anton Evangelatov Date 2026-06-15 Status code-complete on develop, pre-hardfork Ships with Lagoon
The problem

Baseline EVM overcharges for warm state

EIP-2929 prices the first access to an account or storage slot as cold — and every later access in the same transaction as warm.

But warmth resets at each transaction boundary. Tx B pays full cold price for state that tx A already loaded into memory earlier in the same block.

The execution client did the I/O once. Charging the second caller as if it were cold is consensus-correct but economically wasteful.

Cold vs warm (EIP-2929)

AccessColdWarmΔ
Account26001002500
SLOAD21001002000
SSTORE22001002100

SDM rebates exactly this differential — but across transactions within a block, which the EVM never does on its own.

Definition

SDM = Sequencer-Defined Metering

An in-protocol, consensus-critical mechanism for sequencer-defined per-tx gas refunds. The first policy — block-level warming rebates (EIP-7863) — rebates the EIP-2929 cold→warm differential when a tx touches state an earlier tx in the same block already warmed.

Now specified in the Lagoon upgrade: specs/protocol/lagoon/post-exec.md (the generic envelope) + sdm.md (the version-1 schema). SDM is the only defined payload schema today.

Sequencer-defined

The sequencer computes & commits the rebates (in the 0x7D tx). Verifiers re-derive and canonicalize against that commitment.

Inspectable, not re-derived

The consensus payload is just {index, gasRefund} — no provenance. Verifiers apply the claimed value. Off-chain replay re-derives warming & attributes each rebate to its first-warming tx as a sanity check.

Not EIP-3529 capped

It's an adjustment of real work done, not a discretionary refund — so the "≤ 20% of gas used" cap does not apply.

Acceptance goal (DoD): a sequencer can define and apply a custom rebate formula at block production — e.g. a block-level warming discount for Uniswap — running in production on Unichain, with consensus-level verification code living in the monorepo.

The rules

Who gets rebated, and how much

Warm touchRebateWhen

  • Normal user txs claim · deposits warm but never claim · the 0x7D tx never claims
  • Intrinsic warmth excluded — beneficiary, precompiles, access-list & 7702 authorities are already priced warm
  • First-touch provenance — first tx to touch a slot owns it; dedup per tx

Spec validity rules (version 1) — block is invalid if:

  • gasRefundEntries is empty (no refunds → no 0x7D tx at all)
  • any entry has gasRefund == 0
  • entries not strictly increasing by index
  • an index targets a non-standard tx, or refund > evmGasUsed

Each refunded tx's JSON-RPC receipt gains opGasRefund (not in the RLP receipt / trie; null when inactive or no refund).

Conservation of ETH

Where the rebate money comes from

When SDM subtracts Δ gas from a tx, the sender is credited and four recipients are debited — and they sum exactly. Total supply is preserved.

// per refunding tx, applied AFTER EVM execution
gas_delta         = raw_gas - canonical_gas
base_fee_delta    = gas_delta × basefee
beneficiary_delta = gas_delta × (eff_price - basefee)
operator_fee_delta = opfee(raw) - opfee(canonical)

sender_credit = gas_delta × eff_price + operator_fee_delta
= beneficiary_delta + base_fee_delta + operator_fee_delta

When & how it settles

EVM runs with full cold pricing untouched. Settlement applies after the tx's EVM frame, before its state delta commits — atomic with that tx, no separate receipt. Layers onto an unmodified EVM. The L1 fee vault is not adjusted (L1 cost is independent of L2 gas).

Underflow guard

If any recipient can't cover its debit → PostExecSettlementUnderflow and the block fails. (In practice they already collected the fees being refunded.)

The single most important concept

Two layers: PostExec (mechanism) vs SDM (feature)

mechanism PostExec*

A generic, feature-agnostic transport for block-level post-execution data.

  • Synthetic tx type 0x7D + versioned PostExecPayload
  • Executor modes: Disabled / Produce / Verify / Invalid
  • debug_replaySDMBlock RPC + post-exec-replay crate

feature SDM*

Today's only consumer of the PostExec mechanism: the warming-refund computation.

  • SDMWarmingInspector, SDMGasEntry, WarmingRefundKind
  • The opt-in toggle & admin RPCs
  • op-acceptance-tests/tests/sdm/ tooling

The spec formalizes exactly this split: post-exec.md defines the generic envelope (tx type, encoding, structural rules) and is schema-agnostic; sdm.md defines schema version 1. A future policy adds a new version and its own doc — without touching SDM. Rule of thumb: envelope/modes/replay → PostExec*; warming-specific → SDM*.

Wire format

The 0x7D transaction

struct PostExecPayload {
  version: u64,        // = 1
  block_number: u64,   // anchors the tx
  gas_refund_entries: Vec<SDMGasEntry>,
}
struct SDMGasEntry {
  index: u64,          // tx index in block
  gas_refund: u64,     // canonical refund
}
  • Encoding: 0x7D || rlp([version, blockNumber, entries]) — payload not double-wrapped
  • Appended as the last tx in the block; ≤ 1 per block
  • No signature, no signer — no (v,r,s) in the encoding or hash preimage; trusted because the block is. from = 0x0, gas 0
  • Does not execute EVM code — a receipt-bearing metadata envelope
  • No nonce — blockNumber anchors uniqueness (design-doc PoC had a nonce; replaced)
  • Type 0x7D picked high: 0x7E=deposit, 0x7F reserved
  • Never mempool/gossiped — rejected by eth_sendRawTransaction; reaches verifiers only inside a block

Every decode path is version-checked

2718 network/fallback, plain RLP, standalone bytes, reth compact storage — all funnel through decode_checked. Unknown version → "unsupported post-exec payload version". Old nodes loud-fail, which is correct.

RPC shape — go-ethereum friendly

Serializes as a 5-field object: hash, type:0x7d, gas:0, value:0, input:<rlp>. op-node & geth-based clients round-trip it as an opaque typed tx without knowing about PostExec.

Gas bookkeeping

ComponentCounts as
Standard tx icanonicalGasUsed = evmGasUsed − refund
Deposit txevmGasUsed (no refund)
0x7D txzero gas

Block gasUsed / cumulativeGasUsed sum the canonical values. The 0x7D receipt inherits the preceding tx's cumulativeGasUsed, reports success, empty logs, zero bloom.

A single 0x7D as the last tx carries refunds for all txs in the block — there is never more than one, and it's absent entirely when no tx earns a refund.

The heart of it

Four executor modes — chosen up front, fixed for the block

ModeWhen chosenWhat the executor does
Disabled Flag off · pre-hardfork · block has no 0x7D Legacy accounting, no inspector. Byte-identical to pre-feature behavior. A stray 0x7D → fails.
Produce Sequencer, opted in, building a block Inspector runs per tx; at the end drains non-zero refunds, wraps in PostExecPayload, appends the 0x7D tx.
Verify Follower (or self-check) executing a block with a 0x7D Payload is authoritative — canonicalizes each tx against its entry; every entry must be consumed or the block is rejected.
Invalid A 0x7D whose payload didn't decode apply_pre_execution_changes errors immediately. Block rejected.

Consensus model: Verify applies the claimed refunds — it does not re-derive them. Warming reproducibility is an off-chain sanity check (the replay tool), not a consensus determinism rule. This is what makes the policy "sequencer-defined" and what enables configurable rebate formulas later.

Producer side

The warming inspector — first-touch attribution

SDMWarmingInspector is a revm Inspector holding block-scoped maps that persist across all txs:

warmed_accounts: Map<Address, Provenance>
warmed_slots:    Map<(Addr,Slot), Provenance>

At the first top-level frame it records intrinsic warmth (beneficiary, precompiles, access & auth lists) — never rebated. As opcodes touch state, it records first-touch provenance; a touch already in the map → emit_refund.

The composite-inspector fix that matters

Commit 3b7b06fd: previously if the user's inspector short-circuited a call/create, the warming inspector was skipped for that hook → different producers/verifiers could compute different totals → consensus divergence.

Now both run unconditionally; user outcome wins precedence. Invariant: any inspector plugged into the OP EVM must not depend on order-of-invocation with the warming inspector.

Subtlety · confirmed bug

intrinsic warm accounts — and the sender/to over-refund

EIP-2929 pre-loads a tx's accessed-address set before the first opcode: tx.sender, tx.to (or the created address), and the precompiles — EIP-3651 adds the coinbase. Those are intrinsically warm: billed 100, never 2600. No surcharge is paid → no rebate is owed.

The gap

collect_intrinsic_warmth builds the per-tx exclusion set from beneficiary, precompiles, access-list & 7702 authorities — but not sender / to. They're fed in with allow_refund = true instead.

self.collect_intrinsic_warmth(ctx);
let caller = ctx.tx().caller();
self.observe_account_touch(caller, true);   // refundable!
if let TxKind::Call(to) = ctx.tx().kind() {
  self.observe_account_touch(to, true);    // refundable!
}

The tell: the code remembers the newer EIP-3651 coinbase pre-warm but forgot the original EIP-2929 sender+to pre-warms.

  • When it fires: any later tx whose sender or to an earlier tx in the block already warmed gets a spurious +2500.
  • Same-EOA sequential txs (bot / hot wallet / system sender) — every tx after the first.
  • Popular to contract (router, USDC) — 2nd+ tx to it. Directly hits the "warming discount for Uniswap" use case.
  • Not a mint: the three guards still hold (per-tx refund ≤ evmGasUsed, value conservation, underflow). It's mis-metering — vaults/beneficiary leak to senders; consensus-affecting but wrong-but-consistent, not a fork.

Fix

intrinsic_warm_accounts.insert(caller);
if let TxKind::Call(to) = ctx.tx().kind() {
  intrinsic_warm_accounts.insert(to);
}

Keep them in warmed_accounts so a later tx that touches them via a real (genuinely cold) opcode still earns its legitimate rebate. Open: confirm to pre-warm on plain value transfers; handle the Create address.

EL integration

op-reth: the opt-in & the build path

Two-gate activation model

  • Consensus gate — is_sdm_active_at_timestamp (hardfork). Decides whether 0x7D is allowed.
  • Operator opt-in — live AtomicBool, flipped by admin RPC admin_setSdmPostExecOptIn / admin_sdmStatus. Decides whether this sequencer produces them.

Replaced the old static --rollup.sdm-enabled CLI flag. Followers always verify whatever the chain mandates; only production is opt-in.

Produce build path

post_exec_builder_for_next_block(.. Produce)
   ↓  inspector wired per tx
execute all user + deposit txs
   ↓
take_post_exec_entries()  // non-zero only
   ↓
build TxPostExec(block_num, entries)
   ↓
execute_transaction(post_exec)
   ↓  if fails → ABORT build
finish() — 0x7D sealed as last tx

A partial block with refunds applied but no witnessing 0x7D would be unverifiable — so it's never silently dropped.

Builder side

Flashblocks: op-rbuilder + rollup-boost

Test-only path. The op-rbuilder / rollup-boost flashblocks work exists to exercise streaming SDM block-building under test. The production streaming integration is subblocks (roadmap #105, gated on subblocks reaching prod) — not these two components.

Any flashblock must be materializable into a valid block — so intermediate views also need a valid trailing 0x7D.

Wire shape

SDM rides in the flashblock diff, not metadata:

ExecutionPayloadFlashblockDeltaV1 {
  transactions: Vec<Bytes>,   // append-only
  post_exec_tx: Option<Bytes>, // replaced
}

Materialize: base + concat(txs) + latest(post_exec_tx). rollup-boost appends only the latest.

  • op-rbuilder builds the diff & computes roots over the materialized view including the 0x7D
  • rollup-boost folds diffs & appends the latest 0x7D as the final tx
  • Three builders now share build_current_post_exec_tx / materialize_post_exec — standard, flashblocks, and op-reth produce-side rules in one place
  • flashblocks-rpc filters the synthetic tx out of user-facing block/receipt/nonce views

Derivation gate

Post-Lagoon, IsSDM lets batches carry 0x7D. Pre-SDM, a batch with a 0x7D is dropped (batches.go).

EL is the authority

Structural derivation checks were intentionally dropped — the EL validates the embedded payload in Verify mode.

Rebuild paths → Verify

Reorg recovery, window expiry, P2P-less verifier (no_tx_pool) all force-build the derived block in Verify (#21321).

CL & fault proofs

Derivation gating and proof validation

Derivation (op-node / kona-node)

Once SDM is active, span batches legitimately carry 0x7D txs. The IsSDM gate decides acceptance:

// op-node/rollup/toggles.go
func (c *Config) IsSDM(t uint64) bool {
    return c.IsInterop(t)   // rides the toggle
}

Before SDM: a batch with a 0x7D is dropped in batches.go. The EL is the validity authority — structural derivation checks were intentionally dropped.

Fault proofs (kona)

The kona proof executor validates the same invariants under the fault-proof VM: at-most-one, must-be-last, block-number match, no deposit/self targets, refund ≤ raw gas.

Covered by 7 reject-matrix tests in builder/core/tests.rs (landed via #21034). Open: the absolute prestate must be regenerated for the Lagoon hardfork.

Activation

SDM ships with Lagoon

The spec says SDM is "gated by the Lagoon activation timestamp" directly. But all three clients on develop still route through the Interop toggle:

spec: IsSDM(t) → IsLagoon(t)
code: IsSDM(t) → IsInterop(t) → IsLagoon(t)

PR #21105 renamed the Interop hardfork → Lagoon; the Interop toggle was kept separate so the feature can decouple from the fork later. Behaviorally identical today (toggle ⇔ Lagoon).

next Decouple SDM → Lagoon directly (#21129)

SDM and Interop are unrelated features that merely activate at the same fork today. If anyone ever decouples the Interop toggle from Lagoon, IsSDM would silently follow it to the wrong activation — a latent consensus bug.

Fix: gate SDM on IsLagoon / is_lagoon_active directly across op-node, op-reth, kona. Behavior-preserving today; removes the hazard. (still pending on develop)

Operator tooling

Off-chain verification: replay

debug_replaySDMBlock operator-only

RPC on the op-reth node. Replays a block in counterfactual mode against live state and returns synthesized payload, per-tx replay refunds, canonical-vs-raw gas, warming provenance, and mismatches.

Cost: O(block × state), no per-call cap; Semaphore(3) caps concurrency. Hardening (admin namespace + auth + ceilings) is a pre-canary item.

sdmreplay Go CLI

Drives the RPC across a block range, emits JSONL (run_config / block / tx / mismatch / summary).

--fail-on-mismatch → non-zero exit, so it runs as a CI gate or cron monitor.

Detects: duplicate/out-of-range index, deposit/self target, refund>gas, payload≠replay, receipt opGasRefund≠replay.

Reminder: replay is a diagnostic sanity check, not a consensus rule. Consensus applies the claimed refunds; replay confirms a producer's claims match re-derived warming.

Takeaways

Five things to remember

  • SDM rebates cross-tx warm-state access within a block — the EIP-2929 cold→warm differential the EVM never reclaims.
  • PostExec is the mechanism, SDM is the feature. A generic 0x7D envelope; SDM is its first consumer.
  • Consensus applies claimed refunds, doesn't re-derive them — replay is an off-chain sanity check. This enables configurable policies.
  • Code-complete on develop across the whole stack; four consensus-correctness fixes just landed — one open: the sender/to intrinsic-warm over-refund.
  • Ships with Lagoon — remaining work is configurable policy, hardfork-gate decoupling, test hardening, and the devnet→canary→mainnet rollout.
